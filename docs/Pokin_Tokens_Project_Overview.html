1. Executive Summary

Pokin' Tokens is a cross-platform social sports prediction app built for the 2026 FIFA World Cup — the first World Cup hosted by three nations (USA, Canada, Mexico) and the first with 48 teams. The app targets high school students who want to predict match outcomes, challenge friends, and compete on school-based leaderboards using virtual tokens.

Key Differentiators:
  • School-based social competition — Leaderboards segmented by school and state drive organic, word-of-mouth adoption within student communities
  • Virtual token economy — No real money involved, making it school-appropriate and COPPA-friendly
  • AI-powered engagement — Perplexity-powered research lab and Gemini-powered chatbot mascot ("Pokin' Pete")
  • Multi-sport extensibility — Architecture supports expansion beyond soccer (NBA Playoffs already integrated)
  • Premium Gen-Z UI — Dark theme with glassmorphism, animated gradients, and neon accents

Data Flow: Making a Prediction

  1. User selects a match and picks an outcome (e.g., "USA wins")
  2. Flutter sends POST to createPrediction Cloud Function with auth token
  3. Function verifies auth, validates wager against user balance
  4. Function creates prediction doc in Firestore, decrements user tokens atomically
  5. Every 5 minutes, resolveMatchPredictions checks API-Football for completed matches
  6. For each completed match, function evaluates pending predictions
  7. Winners receive wager × multiplier tokens via FieldValue.increment()

Caching Strategy

API-Football has rate limits, so all sports data is cached in Firestore:

CacheTTLRationale
Match fixtures2 minutesNear-real-time for live scores
Standings30 minutesChanges only after match completion

5. Platform Strategy: iOS App vs Web App

How Flutter Enables Multi-Platform

Flutter compiles from a single Dart codebase to three distinct targets:

PlatformBuild CommandOutputDistribution
iOSflutter build iosNative iOS binary (ARM64)Apple App Store / TestFlight
Androidflutter build appbundleAndroid App BundleGoogle Play Store
Webflutter build webHTML/JS/WASMFirebase Hosting / any CDN

iOS App Setup

The project is fully configured for iOS deployment:

  • Xcode project — ios/Runner.xcodeproj and Runner.xcworkspace (CocoaPods)
  • Bundle info — Info.plist with display name "Pokin' Tokens", portrait + landscape orientations
  • Firebase — GoogleService-Info.plist generated via FlutterFire CLI
  • Auth — Apple Sign-In configured (required for App Store apps with social login)
  • Build — Requires macOS with Xcode; archive and upload to App Store Connect

Web App Setup

The web build is a Progressive Web App (PWA) with:

  • index.html — Flutter bootstrap loader, responsive viewport, PWA meta tags
  • manifest.json — PWA manifest for "Add to Home Screen" functionality
  • Icons — Favicon and touch icons for mobile bookmarks
  • Responsive layout — Adapts from mobile (bottom nav) to tablet (side rail) to desktop (extended rail)

Shared vs Platform-Specific Code

100% shared Dart code. There is no platform-specific Dart code. All UI, business logic, navigation, and state management is shared across iOS, Android, and Web. The only platform-specific files are native configuration (Info.plist, AndroidManifest.xml, index.html) which Flutter manages automatically.

7. Security Model

Defense in Depth

Client Layer

  • Firebase Auth tokens required for all API calls
  • No API keys stored in client code
  • Auth state managed by Riverpod with automatic refresh

API Layer

  • Cloud Functions verify Bearer tokens via Firebase Admin SDK
  • All external API keys stored as Firebase Secrets (not environment variables)
  • CORS enabled with domain restrictions

Database Layer

  • Users: Read own doc only; can only update displayName (string, 1-20 chars)
  • Predictions: Read own only; all writes are server-side via Admin SDK
  • Cache: Read-only for authenticated users
  • Default: Deny all unmatched paths
  • Token writes: Server-side only via FieldValue.increment() — users cannot modify their own balance

Business Logic

  • Wager validation: server checks balance before accepting prediction
  • Atomic token operations via Firestore transactions
  • Prediction resolution runs server-side on a schedule (no client trust)
